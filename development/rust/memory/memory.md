# Rust | Управление памятью

Разберем управление памятью в Rust шаг за шагом.
Представь, что данные — это инструменты в мастерской.
Правила Rust просто гарантируют, что два человека не будут одновременно тянуть на себя один и тот же гаечный ключ.

---

## 1. Фундамент: `Box`

**Назначение:** Перенос данных из быстрого, но ограниченного стека в просторную кучу (heap). У данных остается строго один владелец.

```rust
struct BigData {
    data: [u8; 10000],
}
fn main() {
    let my_box = Box::new(BigData { data: [0; 10000] });
    println!("Данные в куче, указатель в стеке: {}", my_box.data[0]);
}

```

* **Как работает:** Box выделяет память в куче. Когда Box выходит из области видимости, память автоматически очищается.
* **Ограничения:** Владелец всегда один. Хочешь отдать — отдаешь навсегда (move) или даешь во временное пользование (borrow).
* **Как лучше использовать:** Для создания рекурсивных структур данных (например, деревьев) или передачи больших объектов без копирования.
* **Как нельзя делать:** Оборачивать в Box мелкие примитивы (вроде i32 или bool), это только замедлит работу из-за лишних обращений к памяти.

## 2. Временное пользование: `&T` и `&mut T`

Это ссылки. Они позволяют смотреть на данные или менять их, не забирая владение у хозяина. Проверяются на этапе компиляции.

### `&T` (Неизменяемая ссылка)

**Назначение:** Чтение данных без права на изменение.

```rust
fn print_length(s: &String) {
    println!("Длина: {}", s.len());
}
fn main() {
    let word = String::from("Мост");
    let ref1 = &word;
    let ref2 = &word;
    print_length(ref1);
    println!("Слово {} все еще здесь, ref2: {}", word, ref2);
}

```

* **Как работает:** Можно создать бесконечное количество таких ссылок одновременно.
* **Как лучше использовать:** Передача аргументов в функции, которые только читают состояние объекта.
* **Как нельзя делать:** Пытаться изменить данные через `&T` или пытаться сохранить ссылку дольше, чем живет исходный объект.

### `&mut T` (Изменяемая ссылка)

**Назначение:** Временное эксклюзивное право на изменение данных.

```rust
fn append_word(s: &mut String) {
    s.push_str(" работает");
}
fn main() {
    let mut word = String::from("Механизм");
    append_word(&mut word);
    println!("{}", word);
}

```

* **Ограничения:** Изменяемая ссылка в один момент времени может быть **только одна**. Никаких других ссылок (ни для чтения, ни для записи) параллельно существовать не может.
* **Как лучше использовать:** Мутация состояния объектов без передачи владения.
* **Как нельзя делать:** Создавать &mut на тот же объект в той же области видимости дважды. Компилятор выдаст ошибку.

## 3. Разделяемое владение (Один поток): `Rc` и `RefCell`

Иногда у данных должно быть несколько полноправных владельцев, и неизвестно, кто закончит работу последним.

### `Rc` (Reference Counted)

**Назначение:** Множественное владение данными только для чтения.

```rust
use std::rc::Rc;
fn main() {
    let shared_config = Rc::new(String::from("db_host=localhost"));
    let worker1 = Rc::clone(&shared_config);
    let worker2 = Rc::clone(&shared_config);
    println!("Конфиг у первого: {}, ссылок: {}", worker1, Rc::strong_count(&shared_config));
}

```

* **Как работает:** Внутри хранится счетчик. При вызове `Rc::clone` счетчик увеличивается. Когда владелец исчезает — уменьшается. При нуле память очищается.
* **Ограничения:** Данные внутри `Rc` **неизменяемы**. Работает только в рамках одного потока.
* **Как нельзя делать:** Пытаться передать `Rc` в другой поток.

### `RefCell`

**Назначение:** Внутренняя мутабельность (изменение данных, даже если внешняя структура считается неизменяемой).

```rust
use std::cell::RefCell;
fn main() {
    let log = RefCell::new(vec!["Старт"]);
    log.borrow_mut().push("Процесс пошел");
    println!("Лог: {:?}", log.borrow());
}

```

* **Как работает:** Переносит проверки правил ссылок (одна мутабельная ИЛИ много иммутабельных) с этапа компиляции на **этап выполнения**.
* **Ограничения:** Если в рантайме ты попытаешься взять две мутабельные ссылки (сделать два borrow_mut() одновременно), программа запаникует и упадет.
* **Как лучше использовать:** Когда логика (и архитектура) требует неизменяемости интерфейса, но внутри нужно вести учет (например, кэширование или счетчики вызовов).

### Комбо: `Rc<RefCell>`

**Назначение:** Несколько владельцев, и каждый может менять данные.

```rust
use std::rc::Rc;
use std::cell::RefCell;
fn main() {
    let shared_state = Rc::new(RefCell::new(0));
    let worker = Rc::clone(&shared_state);
    *worker.borrow_mut() += 10;
    *shared_state.borrow_mut() += 5;
    println!("Итоговое состояние: {}", shared_state.borrow());
}

```

* **Как лучше использовать:** Реализация графов, деревьев с родительскими ссылками или паттерна «Наблюдатель» в однопоточной среде.
* **Как нельзя делать:** Злоупотреблять. Эта конструкция ломает строгую инкапсуляцию. Если можно обойтись явной передачей владения — делай так.

## 4. Разделяемое владение (Многопоточность): `Arc`, `Mutex`, `RwLock`

В многопоточной среде счетчик ссылок должен быть атомарным, а доступ к данным — синхронизированным.

### `Arc` (Atomic Reference Counted)

**Назначение:** Аналог `Rc`, но безопасный для передачи между потоками.

```rust
use std::sync::Arc;
use std::thread;
fn main() {
    let config = Arc::new(String::from("read_only_data"));
    let mut handles = vec![];
    for _ in 0..3 {
        let cfg_clone = Arc::clone(&config);
        handles.push(thread::spawn(move || {
            println!("Поток читает: {}", cfg_clone);
        }));
    }
    for h in handles { h.join().unwrap(); }
}

```

* **Ограничения:** Данные внутри неизменяемы. За атомарность счетчика приходится платить небольшой просадкой производительности по сравнению с `Rc`.

### `Mutex` (Mutual Exclusion)

**Назначение:** Эксклюзивный доступ к данным в многопоточной среде (аналог `RefCell` для потоков).

```rust
use std::sync::{Arc, Mutex};
use std::thread;
fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];
    for _ in 0..5 {
        let counter_clone = Arc::clone(&counter);
        handles.push(thread::spawn(move || {
            let mut num = counter_clone.lock().unwrap();
            *num += 1;
        }));
    }
    for h in handles { h.join().unwrap(); }
    println!("Счетчик: {}", *counter.lock().unwrap());
}

```

* **Как работает:** Чтобы прочитать или изменить данные, поток должен вызвать lock(). Если мьютекс занят другим потоком, текущий поток засыпает и ждет очереди.
* **Как нельзя делать:** Забывать отпускать лок (в Rust он отпускается сам при выходе из области видимости) или пытаться взять лок дважды в одном потоке (приведет к дедлоку — взаимной блокировке).

### `RwLock` (Read-Write Lock)

**Назначение:** Более умный Mutex. Разрешает либо много одновременных читателей, либо строго одного писателя.

```rust
use std::sync::{Arc, RwLock};
use std::thread;
fn main() {
    let data = Arc::new(RwLock::new(100));
    let read_clone1 = Arc::clone(&data);
    let read_clone2 = Arc::clone(&data);
    let write_clone = Arc::clone(&data);
    let r1 = thread::spawn(move || { println!("Читатель 1: {}", *read_clone1.read().unwrap()); });
    let r2 = thread::spawn(move || { println!("Читатель 2: {}", *read_clone2.read().unwrap()); });
    let w1 = thread::spawn(move || {
        let mut d = write_clone.write().unwrap();
        *d += 50;
    });
    r1.join().unwrap(); r2.join().unwrap(); w1.join().unwrap();
}

```

* **Как лучше использовать:** Когда данные читаются сотни раз в секунду, а обновляются раз в минуту. Здесь `RwLock` даст огромный прирост скорости по сравнению с `Mutex`.

## 5. `Weak` для защиты от утечек

У `Rc` и `Arc` есть уязвимость: циклические ссылки. Если объект А ссылается на Б через `Rc`, а Б ссылается на А через `Rc`, их счетчики никогда не опустятся до нуля. Память утечет.

**Назначение:** «Слабая» ссылка. Позволяет смотреть на данные в `Rc`/`Arc`, но не увеличивает основной счетчик владельцев.

```rust
use std::rc::{Rc, Weak};
use std::cell::RefCell;
struct Node {
    parent: RefCell<Weak<Node>>,
}
fn main() {
    let node = Rc::new(Node { parent: RefCell::new(Weak::new()) });
    let weak_ref: Weak<Node> = Rc::downgrade(&node);
    if let Some(strong_ref) = weak_ref.upgrade() {
        println!("Узел еще жив в памяти");
    }
}

```

* **Как работает:** Чтобы использовать данные, слабую ссылку нужно «апгрейднуть» до сильной (upgrade()). Она вернет Option, так как данные к этому моменту могут быть уже удалены.
* **Как лучше использовать:** Связи типа «ребенок -> родитель» в деревьях, графах или кэшах, где дочерние элементы не должны удерживать родителя в памяти насильно.

> **Помни о тестировании!** Все эти конструкции компилируются успешно, но `RefCell` и мьютексы могут вызвать панику, дедлок и гонку данных в рантайме при нарушении логики. Их необходимо покрывать тестами в первую очередь.
