# Философия надежного многопоточного кода

## Введение

Создание многопоточного кода — это как проектирование сложного часового механизма. Главная ценность здесь не в том, чтобы заставить все шестеренки вращаться на предельной скорости любой ценой, а в том, чтобы система оставалась предсказуемой, безопасной и легко читаемой.

Правильный подход опирается на строгую инкапсуляцию. Каждый объект должен сам отвечать за свою потокобезопасность и целостность контракта. Внешний код, который вызывает методы твоего объекта, не должен знать, скрыт ли внутри мьютекс, канал или сложная lock-free логика. Разделение ответственности означает, что бизнес-правила не должны перемешиваться с низкоуровневым жонглированием блокировками. Производительность важна, но она всегда строится поверх железобетонной безопасности: самый быстрый алгоритм теряет смысл, если он изредка выдает случайные, невоспроизводимые ошибки.

---

## 1. Базовые понятия и проблемы (Словарь)

* **Race Condition (Состояние гонки):** Ситуация, когда результат работы программы зависит от того, в каком порядке планировщик ОС выделил процессорное время потокам. Если два потока одновременно читают `x = 5`, прибавляют 1 и записывают обратно, вместо `7` ты получишь `6`.
* **Deadlock (Взаимная блокировка):** Поток А захватил Ресурс 1 и ждет Ресурс 2. Поток Б захватил Ресурс 2 и ждет Ресурс 1. Программа висит вечно.
* **Memory Visibility (Видимость памяти):** Под капотом процессор кэширует данные в L1/L2 кэшах каждого отдельного ядра. Если поток на Ядре 1 изменил переменную, поток на Ядре 2 может продолжать читать старое значение из своего кэша, пока архитектура памяти не получит явную команду на синхронизацию (memory barrier).
* **Livelock (Динамическая взаимоблокировка):** Потоки не зависают полностью (как при Deadlock), а постоянно меняют свое состояние в ответ на действия друг друга, не выполняя полезной работы. Это как два вежливых человека в узком коридоре, которые бесконечно стараются уступить друг другу дорогу.
* **Starvation (Голодание потоков):**  Ситуация, когда поток постоянно оттесняется другими, более «агрессивными» или высокоприоритетными потоками, и не может получить доступ к ресурсу для выполнения своей задачи.

---

## 2. Главные архитектурные подходы

Прежде чем браться за блокировки, стоит рассмотреть более чистые альтернативы:

* **Неизменяемость (Immutability):** Фундамент надежного ООП. Объект инициализируется один раз через конструктор и больше не меняет своего состояния. Все методы, требующие изменения логического состояния, возвращают новый экземпляр объекта. Такие объекты абсолютно безопасны для любых потоков по умолчанию.
* **Message Passing (Каналы/Актёры):** Вместо того чтобы потоки ходили в общую память, они общаются, пересылая друг другу неизменяемые сообщения (события). "Не общайтесь, разделяя память; разделяйте память, общаясь".
* **Блокировки (Pessimistic Concurrency):** Поток монопольно захватывает доступ к данным, делает работу, отпускает. Надежно, но может стать бутылочным горлышком.
* **Lock-free (Optimistic Concurrency):** Поток готовит изменения в стороне, а затем пытается применить их одной неделимой (атомарной) операцией процессора. Если кто-то успел изменить данные до него — поток повторяет попытку.

## 3. Примитивы синхронизации (Классика)

Это инструменты контроля доступа, когда без общего состояния не обойтись.

* **Mutex (Mutual Exclusion):** Светофор. Только один поток может владеть мьютексом в моменте времени. Остальные ждут своей очереди (засыпают на уровне ОС).
* **RwLock (Read-Write Lock):** Оптимизированный мьютекс. Позволяет любому количеству потоков читать данные одновременно, но для записи блокирует доступ всем (и читателям, и другим писателям). Полезен, когда чтений в сотни раз больше, чем записей.
* **Semaphore (Семафор):** Счетчик разрешений. Полезен для ограничения пула ресурсов (например, не больше 10 одновременных подключений к базе данных).
* **Condvar (Condition Variable / Барьеры):** Механизм, позволяющий потоку заснуть и проснуться только тогда, когда другой поток подаст сигнал о том, что какое-то условие выполнилось (например, в очередь поступили новые данные).
* **Spinlock (Крутящаяся блокировка):** Поток не засыпает в ожидании ресурса (отдавая контекст ОС), а крутится в бесконечном пустом цикле, постоянно проверяя, не освободился ли замок. Имеет смысл только на многоядерных системах для защиты сверхкоротких операций, чтобы сэкономить такты на переключении контекста ОС.
* **Barrier (Барьер):** Точка синхронизации, в которой потоки обязаны дождаться друг друга. Программа не пойдет дальше, пока все заявленные потоки (например, 4 рабочих потока) не достигнут этого барьера.
* **TLS (Thread-Local Storage / Локальное хранилище потока):** Инструмент для полного избегания синхронизации. Каждому потоку выдается своя независимая копия переменной или объекта. Отлично подходит для изоляции кэшей, генераторов случайных чисел или сборщиков локальной статистики.

---

## 4. Lock-free и атомарные сущности (Хирургия)

Lock-free алгоритмы не используют `Mutex`. Они опираются на аппаратные инструкции процессора.

* **Атомарные типы (`AtomicUsize`, `AtomicBool`, `AtomicPtr`):** Обертки над примитивами, чтение и запись которых процессор гарантированно выполняет за один такт, не давая другим ядрам вклиниться.
* **CAS (Compare-And-Swap):** Сердцевина любого lock-free алгоритма. Это процессорная инструкция (например, `CMPXCHG` в x86), которая делает следующее: "Посмотри на ячейку памяти. Если там лежит значение X, запиши туда Y и верни успех. Если там не X, ничего не трогай и верни ошибку".

## 5. Lock-free коллекции и подходы

Lock-free структуры — это территория сложных алгоритмических компромиссов. Писать их с нуля в продакшн без крайней нужды не стоит, лучше брать проверенные индустриальные реализации.

* **Concurrent Queue (Очереди Майкла-Скотта / Michael-Scott Queue):**  Классическая lock-free очередь на связанных списках. Использует CAS-операции для атомарного обновления указателей головы и хвоста. Идеальна для распределения задач.
* **Ring Buffer (Кольцевой буфер):** Массив фиксированного размера. В вариантах SPSC (Single-Producer, Single-Consumer) можно реализовать абсолютно lock-free взаимодействие без единой CAS-операции, опираясь только на атомарные барьеры памяти (Memory Ordering). Это предел скорости для передачи потоковых данных.
* **SkipList (Список с пропусками):** Lock-free альтернатива стандартным Map и Set (сбалансированным деревьям). Данные хранятся в многоуровневых связанных списках. Позволяет потокам искать, вставлять и удалять элементы одновременно без глобальных блокировок.
* **Epoch-based Reclamation (Управление памятью через эпохи):** В lock-free структурах возникает проблема ABA: пока один поток читает узел, другой может его удалить, освободить память и создать новый узел на том же месте. Чтобы избежать падения, память удаленных элементов не освобождается сразу, а ждет смены «эпохи» — момента, когда ни один активный поток точно не смотрит на старые данные.

## 6. Справочник по инструментам синхронизации

Ниже всё инструменты от самых быстрых архитектурных решений (где потоки вообще не мешают друг другу) до тяжелых системных блокировок. Внутри каждой группы элементы отсортированы по убыванию производительности.

### Группа 1: Изоляция и передача сообщений (Без гонок)

Лучшая синхронизация — та, которой нет. Здесь потоки работают со своими локальными данными или передают право владения объектом без разделения памяти.

| Инструмент | Промышленные реализации (Crates) | Где применять (Сценарий) | Производительность |
| --- | --- | --- | --- |
| **TLS** (Thread-Local Storage) | Встроено: `std::thread_local!` | Изолированные кэши, генераторы случайных чисел, сбор метрик внутри одного потока до их агрегации. | **Максимальная.** Накладные расходы равны нулю, гонки невозможны. |
| **Ring Buffer** (SPSC буфер) | `rtrb`, `ringbuf` | Передача непрерывного потока данных (аудио, телеметрия, пакеты) от одного источника строго одному потребителю. | **Сверхвысокая.** Wait-free. Опирается только на барьеры памяти, без циклов `CAS`. |
| **Каналы** (Channels) | `kanal` (топ для MPMC), `flume`, `crossbeam-channel` | Паттерны Worker-Pool, Actor Model, Producer-Consumer. Диспетчеризация задач и сбор логов. | **Высокая.** Амортизированная стоимость отправки. `kanal` работает на уровне лучших lock-free очередей. |

### Группа 2: Lock-free и Атомики (Оптимистичная конкуренция)

Работают на уровне аппаратных инструкций процессора. Потоки не засыпают, а пытаются протолкнуть изменения. Идеально для участков кода, где задержка (latency) важнее всего.

| Инструмент | Промышленные реализации (Crates) | Где применять (Сценарий) | Производительность |
| --- | --- | --- | --- |
| **Атомарные типы** | Встроено: `std::sync::atomic`, крейт `portable-atomic` | Флаги остановки сервисов, счетчики статистики, метрики, стейт-машины простого железа. | **Сверхвысокая.** 1-2 такта процессора при слабой конкуренции. Падает при шквале одновременных записей (cache trashing). |
| **Spinlock** | `spin`, встроено как часть `parking_lot::Mutex` | Защита микро-операций (выделение памяти, обновление указателя). | **Высокая (но опасная).** Эффективно только если захват длится наносекунды. При долгом удержании сжигает процессор впустую. |
| **Concurrent Queue** | `crossbeam-queue` (SegQueue, ArrayQueue) | Неблокирующие очереди задач для пула потоков. | **Высокая.** Используют циклы `CAS`. Быстрее мьютексов, но сложнее в отладке. |
| **SkipList** (Map / Set) | `crossbeam-skiplist` | Конкурентные словари. Быстрый поиск маршрутов, кэширование ключей без блокировки всей таблицы. | **Отличная для чтения.** Чтение идет параллельно без `CAS`, запись использует `CAS` и `Epoch-based` сборку мусора. |
| **Epoch Reclamation** | `crossbeam-epoch` | Основа для создания собственных lock-free структур. Безопасное удаление узлов в памяти (решение проблемы ABA). | **Высокая.** Позволяет памяти освобождаться автоматически, не прерывая работу читателей. |

### Группа 3: Блокировки (Пессимистичная конкуренция)

Поток обращается к ядру операционной системы. Если ресурс занят, ОС усыпляет поток и будит его позже. Это спасает процессор от холостого хода, но переключение контекста стоит дорого.

| Инструмент | Промышленные реализации (Crates) | Где применять (Сценарий) | Производительность |
| --- | --- | --- | --- |
| **RwLock** | `parking_lot::RwLock` | Глобальные конфигурации, справочники, таблицы маршрутизации. Редкая запись, очень частое чтение. | **Средняя.** Читатели не блокируют друг друга. Падает до уровня `Mutex`, если записей становится много. |
| **Mutex** | `parking_lot::Mutex` | Универсальная защита любых общих структур данных (векторы, деревья, сложные бизнес-сущности). | **Средне-Низкая.** Тяжелое переключение контекста ОС. `parking_lot` быстрее `std` за счет умного короткого spin-цикла перед засыпанием. |
| **Семафор** | `tokio::sync::Semaphore` (для async), `sema` (sync) | Ограничение числа подключений к БД, лимитирование одновременных тяжелых вычислений. | **Низкая.** Используется не для скорости, а для защиты внешних ресурсов от перегрузки (Rate Limiting). |
| **Condvar / Barrier** | `parking_lot::Condvar`, `std::sync::Barrier` | Синхронизация стадий инициализации системы. Поток ждет, пока накопится порция данных для обработки. | **Низкая.** Инструменты оркестрации, созданные специально для того, чтобы потоки спали. |

## Конкурентные коллекции

Конкурентные коллекции требуют особой инженерной дисциплины. Динамическое выделение памяти (например, при расширении массива) в многопоточной среде — это всегда риск задержек или скрытых гонок. Поэтому золотое правило: если можно использовать неизменяемые (immutable) структуры и передавать их копии, так и нужно делать.

Если же разделяемое изменяемое состояние неизбежно (например, глобальный кэш или очередь задач), мы опираемся на промышленные решения. Ниже приведена классификация для экосистемы Rust, отсортированная по производительности внутри каждой группы.

### 1. Словари и Множества (Maps & Sets)

Самая востребованная категория для хранения сессий, кэшей и реестров объектов.

| Библиотека (Крейт) | Механика под капотом | Производительность | Особенности и применение |
| --- | --- | --- | --- |
| **`scc::HashMap`**<br>

<br>`scc::TreeIndex` | Lock-free, основано на атомарных указателях и батчинге. | **Сверхвысокая** при интенсивной записи. | **Применение:** Высоконагруженные реестры (например, учет миллионов соединений или тегов оборудования).<br>

<br>**Фичи:** Одна из самых быстрых реализаций в мире. Безопасно масштабируется без "stop-the-world" блокировок при изменении размера хэш-таблицы. Имеет асинхронный API. |
| **`dashmap`** | Sharded Locks (Шардирование блокировок). Таблица разбита на сегменты, каждый защищен своим `RwLock`. | **Высокая.** Де-факто индустриальный стандарт. | **Применение:** Общие in-memory кэши, словари конфигураций.<br>

<br>**Фичи:** Очень простой API, почти идентичный стандартному `HashMap`. Идеален, когда потоки чаще обращаются к *разным* ключам (нет конфликтов за один сегмент). |
| **`crossbeam-skiplist`**<br>

<br>(SkipMap / SkipSet) | Lock-free структура на многоуровневых связанных списках. | **Отличная** для чтения, **Средняя** для записи (O(log n)). | **Применение:** Когда ключи должны быть всегда отсортированы (например, хранение событий по временным меткам, биржевые стаканы).<br>

<br>**Фичи:** Позволяет lock-free итерацию по диапазонам ключей, что невозможно в обычных хэш-мапах. |
| **`im::HashMap`** | Immutable Data Structure (Персистентные структуры на основе деревьев). | **Отличная** при редких записях, нет блокировок при чтении. | **Применение:** Архитектуры со строгим разделением состояния. Настройки систем.<br>

<br>**Фичи:** Клонирование такой коллекции занимает O(1). При изменении создается новая версия, переиспользующая 99% старой памяти. Полностью исключает гонки. |

### 2. Очереди и Буферы (Queues & Ring Buffers)

Инструменты для передачи данных между компонентами. Здесь важна пропускная способность (throughput) и минимальная задержка (latency).

| Библиотека (Крейт) | Механика под капотом | Производительность | Особенности и применение |
| --- | --- | --- | --- |
| **`rtrb`**<br>

<br>(Ring Buffer) | Wait-free, SPSC (Single-Producer, Single-Consumer). Только барьеры памяти. | **Максимальная.** Без циклов CAS и системных вызовов. | **Применение:** Непрерывный поток сырых данных. Снятие показаний с АЦП, энкодеров, аудио-потоки.<br>

<br>**Фичи:** Идеально ложится на кэш-линии процессора. Выделяет память один раз при старте. |
| **`kanal`** | Lock-free внутри, оптимизированный memory-layout для MPMC (Multi-Producer, Multi-Consumer). | **Высокая.** Обходит стандартные каналы и `crossbeam` в бенчмарках. | **Применение:** Диспетчеризация задач, сбор логов со всех модулей системы в один писатель.<br>

<br>**Фичи:** Одинаково хорошо работает как в синхронном, так и в асинхронном (`tokio`) коде. |
| **`crossbeam-queue`**<br>

<br>(ArrayQueue) | Lock-free MPMC на основе массива фиксированного размера. | **Высокая.** Опирается на `CAS` операции. | **Применение:** Распределение работы в пуле потоков (Worker pool), где размер очереди заранее известен и ограничен. |
| **`crossbeam-deque`** | Work-stealing очередь. У каждого потока своя очередь, но они могут "воровать" задачи у соседей. | **Высокая.** Минимизирует конкуренцию (contention). | **Применение:** Планировщики задач, тяжелые вычисления, разбитые на сотни мелких подзадач (используется под капотом `rayon`). |

### 3. Списки и Массивы (Vectors)

Вектор — самая сложная структура для многопоточности. Динамическое увеличение емкости (reallocation) требует переноса всех данных в новую область памяти, что крайне сложно сделать без блокировки всех читателей.

| Библиотека (Крейт) | Механика под капотом | Производительность | Особенности и применение |
| --- | --- | --- | --- |
| **`scc::Vector`** | Конкурентный вектор на основе фрагментированных массивов. | **Высокая.** | **Применение:** Журналы только для добавления (append-only logs), конкурентный сбор плоских метрик.<br>

<br>**Фичи:** Позволяет параллельно добавлять элементы (`push`) без инвалидации ссылок у читающих потоков. |
| **`im::Vector`** | Immutable B-Tree вектор. | **Высокая** для клонирования и чтения. | **Применение:** Снимки (snapshots) состояния системы. Передача списка параметров в рабочие потоки.<br>

<br>**Фичи:** Изменение элемента в середине массива не требует копирования всего массива, только перестроения ветки дерева. |
| **`Mutex<Vec<T>>`**<br>

<br>`RwLock<Vec<T>>` | Блокировка всего массива на уровне ОС. | **Низкая** при высокой конкуренции. | **Применение:** Простые буферы накопления, где запись происходит редко (или большими пачками), а чтение быстрое.<br>

<br>**Фичи:** Легко читать, легко сопровождать. Если нет доказанных проблем с производительностью, начинать стоит именно с этого. |

## 8. Практика: Проектирование структур

Строго инкапсулируй логику синхронизации внутри объекта. Внешний код не должен знать, как именно структура обеспечивает потокобезопасность. Иными словами структура скрывает внутри один RwLock или lock-free коллекцию, а наружу предоставляет четкие бизнес-методы (например, update_sensor_data(), get_current_status()). Это гарантирует целостность объекта и его потокобезопасность.

### 8.1 Непрерывный поток метрик или событий

* **Задача:** Множество датчиков или компонентов системы генерируют события (телеметрию, логи). Нужно собирать их и обрабатывать, не замедляя работу отправителей.
* **Решение:** Message Passing (Паттерн Producer-Consumer). Создается lock-free канал (MPSC — Multi-Producer, Single-Consumer). Компоненты-источники получают независимые объекты-передатчики, а отдельный фоновый поток вычитывает приемник.

```rust
use std::sync::mpsc::{channel, Sender, Receiver};
pub struct TelemetryTransmitter {
    sender: Sender<f64>,
}
impl TelemetryTransmitter {
    pub fn new(sender: Sender<f64>) -> Self {
        Self { sender }
    }
    pub fn dispatch(&self, value: f64) {
        let _ = self.sender.send(value);
    }
}

```

### 8.2 Высокоинтенсивный сбор статистики

* **Задача:** Нужно считать количество проходов цикла, тики энкодера или ошибки на лету. Обновлений миллионы в секунду, операции элементарные (инкремент).
* **Решение:** Атомарные типы (`AtomicUsize`). Использование `Mutex` здесь убьет производительность из-за постоянных блокировок операционной системы, а атомарный инкремент выполняется за один такт процессора.

### 8.3 Общий справочник конфигураций

* **Задача:** В памяти хранится таблица маршрутизации или настройки, к которым ежесекундно обращаются сотни потоков. Обновление справочника происходит редко (раз в минуту).
* **Решение:** `RwLock`. Обычный `Mutex` заставил бы читающие потоки выстраиваться в очередь. `RwLock` позволяет им читать данные параллельно, блокируя систему только на краткий миг для редкой перезаписи конфига.

### 8.4 Изоляция состояния для сложной математики

* **Задача:** Потоку нужно выполнить тяжелый расчет (например, сведение массивов или построение графа), требующий временных буферов и промежуточного состояния.
* **Решение:** Thread-Local Storage или создание независимых объектов внутри потока. Поток собирает все необходимые данные, отключается от общего мира, делает вычисления в полной изоляции и только в конце публикует готовый иммутабельный результат через канал или блокировку.

### 8.5 Простая потокобезопасная структура через Mutex

Подходит для большинства бизнес-задач. Код читаем и надежен.

```rust
use std::sync::Mutex;
pub struct SafeSensorBuffer {
    data: Mutex<Vec<f64>>,
}
impl SafeSensorBuffer {
    pub fn new() -> Self {
        Self { data: Mutex::new(Vec::new()) }
    }
    pub fn push_reading(&self, value: f64) {
        let mut buffer = self.data.lock().unwrap();
        buffer.push(value);
    }
    pub fn average(&self) -> Option<f64> {
        let buffer = self.data.lock().unwrap();
        if buffer.is_empty() {
            return None;
        }
        let sum: f64 = buffer.iter().sum();
        Some(sum / buffer.len() as f64)
    }
}

```

### 8.1 Lock-free структура (Счетчик через CAS-цикл)

Используется там, где борьба идет за наносекунды (например, энкодеры двигателей или сбор высокочастотной телеметрии).

```rust
use std::sync::atomic::{AtomicUsize, Ordering};
pub struct LockFreeCounter {
    count: AtomicUsize,
}
impl LockFreeCounter {
    pub fn new(initial: usize) -> Self {
        Self { count: AtomicUsize::new(initial) }
    }
    pub fn increment(&self) {
        let mut current = self.count.load(Ordering::Relaxed);
        loop {
            let next = current + 1;
            // Пытаемся подменить current на next. Если кто-то успел изменить
            // count между load и compare_exchange, мы получим актуальное
            // значение в actual и пойдем на новый круг.
            match self.count.compare_exchange_weak(current, next, Ordering::SeqCst, Ordering::Relaxed) {
                Ok(_) => break,
                Err(actual) => current = actual,
            }
        }
    }
    pub fn get(&self) -> usize {
        self.count.load(Ordering::SeqCst)
    }
}

```
