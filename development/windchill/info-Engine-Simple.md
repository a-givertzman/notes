# Выборка номенклатуры напрямую через Info*Engine задачу

---

## Разбор механики Info*Engine

Когда вы вызываете вебджект `Query-Objects`, Info*Engine не просто делает выборку из таблицы.
Внутри ядра Windchill этот вызов транслируется в низкоуровневый объект `wt.query.QuerySpec`,
который строит параметризованный SQL-запрос с учетом прав доступа текущего пользователя (Access Control Control lists, ACL).

Чтобы отфильтровать номенклатуру, нам нужно задействовать три ключевых механизма:

1. **Идентификация типа (`TYPE`):** Для деталей это базовый класс `wt.part.WTPart`. Если есть свои подтипы деталей, можно указывать их внутренние имена (например, `wt.part.WTPart|com.company.CustomPart`).
2. **Контейнерная привязка (`containerReference.key.id`):** В архитектуре Windchill проекты, библиотеки и изделия являются контейнерами (`WTContainer`). Каждая деталь хранит жесткую ссылку на свой контекст.
3. **Временные метки (`thePersistInfo.createStamp`):** Системное поле, отражающее точное время записи объекта в базу данных. По нему можем строить фильтр по времени.

## Пошаговая настройка и внедрение Info*Engine задачи

Задачи должны размещаться в специализированной директории, изолированной от обновлений PTC.

### Шаг 1. Создание файла конфигурации задачи

Создайте файл по пути `<Windchill>/tasks/ext/custom/get_project_nomenclature.xml`.

Убедитесь, что внутри кода отсутствуют пустые строки, так как парсер Info*Engine чувствителен к структуре документа.

```xml
<%@page language="java" session="false" encoding="UTF-8"%>
<%@taglib uri="http://www.ptc.com/infoengine/taglib/core" prefix="ie"%>
<ie:webject name="Query-Objects" type="OBJ">
  <ie:param name="INSTANCE" data="$(properties[0]wt.federation.ie.VMName[0])"/>
  <ie:param name="TYPE" data="wt.part.WTPart"/>
  <ie:param name="WHERE" data="(containerReference.key.id=$(@FORM[]projectId[0])&amp;thePersistInfo.createStamp&gt;=$(@FORM[]startDate[0]))"/>
  <ie:param name="VERSION" data="LATEST"/>
  <ie:param name="ITERATION" data="LATEST"/>
  <ie:param name="ATTRIBUTE" data="number,name,obid,thePersistInfo.createStamp,containerReference.key.id" delim=","/>
  <ie:param name="GROUP_OUT" data="project_nomenclature"/>
</ie:webject>

```

### Шаг 2. Как понять, что именно указывать в параметрах

* **Где искать имена атрибутов:** Базовые атрибуты (`name`, `number`, `obid`) стандартизированы. Если вам требуются специфические или пользовательские параметры, используйте утилиту **Типы и атрибуты (Type and Attribute Management)** в панели администрирования Windchill, либо откройте локальный Javadoc (`<Windchill>/codebase/docs/api/index.html`) для анализа структуры классов.
* **Синтаксис фильтрации `WHERE`:** Info*Engine использует логические операторы `&` (И) и `|` (ИЛИ). Обратите внимание, что в XML-конфиге символы амперсанда и «больше/меньше» обязательно экранируются как `&amp;`, `&gt;` и `&lt;`.
* **Динамические переменные:** Конструкция `$(@FORM[]paramName[0])` указывает интерпретатору взять значение первого элемента массива параметров, переданных в HTTP-запросе к задаче.

### Шаг 3. Управление выходным форматом

По умолчанию сервер возвращает XML.
Переключение Info*Engine на генерацию структурированного JSON происходит на уровне протокола вызова.

При отправке HTTP-запроса «соседней системой» достаточно передать заголовок `Accept: application/json` либо дописать в URL параметр `&accept=application/json`.
Серверный сериализатор автоматически трансформирует внутреннюю группу объектов `project_nomenclature` в JSON-массив.

## Клиентская реализация на стороне внешней системы

Ниже представлен пример объектно-ориентированного клиента на языке Python для выполнения этого запроса. Код спроектирован по принципам строгой инкапсуляции.

```python
import requests
class WindchillClient:
    """Клиент для выполнения специализированных интеграционных запросов к Info*Engine."""
    def __init__(self, base_url: str, auth: tuple):
        """
        Инициализирует экземпляр клиента.
        :param base_url: Базовый URL сервера Windchill (например, 'http://server')
        :param auth: Кортеж авторизации (логин, пароль)
        """
        self._base_url = base_url
        self._auth = auth
    def fetch_nomenclature(self, project_id: str, start_date: str) -> list:
        """
        Запрашивает список деталей по ID проекта и дате начала.
        :param project_id: Внутренний числовой идентификатор проекта в Windchill
        :param start_date: Дата в формате UTC (например, '2026-06-01 00:00:00')
        :return: Список словарей с атрибутами деталей
        """
        url = f"{self._base_url}/Windchill/servlet/IE/tasks/ext/custom/get_project_nomenclature.xml"
        payload = {"projectId": project_id, "startDate": start_date, "accept": "application/json"}
        response = requests.post(url, data=payload, auth=self._auth, verify=False)
        response.raise_for_status()
        return response.json().get("project_nomenclature", [])

```

## Тестирование системы

Перед запуском в промышленную эксплуатацию необходимо провести комплексную проверку решения:

1. **Тест на пустую выборку:** Передайте заведомо несуществующий ID проекта. Система должна вернуть статус `200 OK` со ссылкой на пустую группу объектов, не вызывая критических исключений.
2. **Проверка формата дат:** Убедитесь, что формат даты (`YYYY-MM-DD HH:MM:SS`) корректно парсится базой данных. Различия в региональных настройках операционной системы сервера могут приводить к ошибкам SQL-синтаксиса.
3. **Нагрузочный тест:** Запустите выборку по крупному проекту (более 5 000 элементов) и замерьте время ответа. Это позволит определить, укладывается ли запрос в стандартный таймаут прокси-сервера (обычно 60 секунд).

## Держи меня в курсе

При детальном проектировании этой задачи стоит обратить внимание на следующие нюансы, которые могут потребовать доработки в будущем:

* **Постраничный вывод (Пагинация):** Если номенклатура проекта превысит несколько десятков тысяч объектов, один тяжелый запрос перегрузит оперативную память. Рекомендуется сразу заложить в вебджект параметры `PAGE_LIMIT` и `PAGE_OFFSET` для порционной выгрузки.
* **Пользовательские атрибуты (IBA):** Стандартный вебджект `Query-Objects` вытаскивает только жестко смоделированные в Java-классе поля (Hard attributes). Если ваши проекты используют гибкие пользовательские атрибуты (Менеджер типов), их придется извлекать с помощью дополнительного вебджекта `Query-Attributes`, связав их по `obid`.
